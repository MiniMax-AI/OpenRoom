# App Definition Rules

Defines the VibeApp runtime model, lifecycle, data storage, and code structure.

---

## 1. Runtime Model: iframe Micro-Frontend

Each App runs in an independent iframe with its own JS runtime, DOM, and style scope. Apps are completely isolated and communicate with the host via `postMessage` through APIs wrapped by `@/lib` (see `data-interaction.md`).

---

## 2. Application Lifecycle

```text
CREATED → LOADING → DOM_READY → LOADED → REGISTERING → REGISTERED → READY
                                   ↓ ERROR → UNLOADING → DESTROYED
```

Apps only need to report: `DOM_READY` (DOM mounted), `READY` (interactive), `ERROR` (unrecoverable error).

```typescript
import { reportLifecycle } from '@/lib';
import { AppLifecycle } from '@gui/vibe-container';

useEffect(() => {
  reportLifecycle(AppLifecycle.DOM_READY);
  initData()
    .then(() => reportLifecycle(AppLifecycle.READY))
    .catch((err) => reportLifecycle(AppLifecycle.ERROR, err.message));
}, []);
```

**Rules**: Only report in `index.tsx` | `DOM_READY` before `READY` | `ERROR` must include error message

## 3. Data Storage Specification

```text
/                          # App storage root (lib automatically maps NAS path)
├── data/{collection}/{id}.json   # Collection data, ID generated by generateId()
│   └── state.json         # Optional, temporary state file
└── assets/                # Static assets
```

**Collection data**: One JSON file per entity, filename is the ID, Repository Pattern wraps CRUD.

**state.json** (optional): Stores temporary state (playback progress, drafts, preferences, etc.), used for session recovery and cross-device sync.
- App should start normally when state.json does not exist (fall back to default state)
- Agent can modify App runtime state by writing to state.json
- When state.json is enabled, **`SYNC_STATE` Action must be implemented** (see `data-interaction.md` Section 2.6)

---

## 4. App Code Structure

```text
src/pages/{AppName}/
├── components/        # UI Components
├── pages/             # Sub-pages (H5 templates, optional)
├── store/             # State management (Context/Reducer)
├── actions/           # Action definitions and handlers
├── styles/            # CSS Variables / CSS Modules
├── i18n/              # Multi-language: index.ts + en.ts + zh.ts
├── mock/              # Local mock data (TypeScript exports)
├── data/              # Seed data (pure JSON, deployed to NAS)
├── meta/              # Metadata: guide.md + meta.yaml (bilingual)
├── index.tsx          # Entry (lifecycle reporting here only)
├── index.module.scss  # Entry styles
└── types.ts           # TypeScript type definitions
```

**Rules**:
- **App must not display its own name internally**; the host system displays it
- `data/` only contains pure JSON | `mock/` contains TypeScript mock data | `types.ts` goes in the root
- `repositories/` wraps file operations; components do not call `@/lib` directly
- `actions/` defines externally callable methods exposed by the App, with retry mechanisms

---

## 5. Multi-Language (i18n) Specification

**No user-visible text should be hardcoded**; all text is output through the `t()` translation function.

- Language is automatically synced to i18n by `fetchVibeInfo()`; manual `i18n.changeLanguage()` is prohibited
- Must support at least `en.ts` and `zh.ts`
- Translation keys use nested namespaces (e.g., `player.play`)
- Logs/console are not restricted

```typescript
// i18n/en.ts
export default { player: { play: 'Play', pause: 'Pause' } };

// In components
const { t } = useTranslation('{appName}');
return <button>{t('player.play')}</button>;
```
